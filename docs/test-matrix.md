# 测试与验收基线

本文档服从 [MVP 总契约](./mvp-contract.md)，不定义新的产品模式，只把 `apps/weapp/test/**`、`services/**/test/**` 的测试分层、验收矩阵和回归优先级写死。  
本文档的直接受众是测试实现者、代码评审者和负责验收的人。它不负责页面交互设计，也不负责采集实现细节；这两部分分别服从 `frontend-contract.md` 和 `ingest-contract.md`。

## 1. 文档目的

测试和验收必须以本文档为唯一基线，避免：

- 页面测试只测展示、不测分流
- 采集测试只测 happy path、不测回退
- SLA 和准确率没有统一口径
- 回归优先级每次临时决定

## 2. 测试分层

测试分层固定为：

1. 共享规则测试
2. 小程序页面测试
3. API 测试
4. ingest 流水线测试
5. 回归测试

对应目录固定为：

- 共享规则测试
  - `packages/shared/test/**`
- 小程序页面测试
  - `apps/weapp/test/**`
- API 测试
  - `services/api/test/**`
- ingest 流水线测试
  - `services/ingest/test/**`
- 回归测试
  - 以上目录中围绕 P0/P1/P2 的稳定用例集合

## 3. 页面行为矩阵

页面矩阵至少覆盖以下行为：

| 编号 | 场景 | 最低要求 |
| --- | --- | --- |
| UI-01 | 首页摘要展示 | 最新公告、来源状态、对比入口可见 |
| UI-02 | 公告详情到岗位列表跳转 | 结构化公告可进入岗位列表 |
| UI-03 | 岗位列表加入对比组 | 可单个加入、批量加入 |
| UI-04 | 跨考试类型禁止对比 | 不允许把不同 `examType` 的岗位放进同组 |
| UI-05 | 满组方案处理 | 触发复用、替换或 review-needed 流程 |
| UI-06 | 人工纠错待核对 -> 岗位页 | 按钮动作正确，路由正确 |
| UI-07 | 可信度待核对 -> 来源状态/复核中心 | 按 `buildTrustAction` 分流 |
| UI-08 | 分享文案 | 包含 triage 总结和待核对拆分 |

页面测试必须优先覆盖：

- `pages.test.js`
- compare 页
- source-status 页
- review-center 页
- positions 页
- notice-detail 页

## 4. 采集链路矩阵

采集矩阵至少覆盖以下行为：

| 编号 | 场景 | 最低要求 |
| --- | --- | --- |
| ING-01 | 单来源失败不拖垮全局 | 其他来源继续运行 |
| ING-02 | 解析失败时回退稳定版 | 前台继续读旧稳定快照 |
| ING-03 | `attachment-only` 进入公告模式 | 不开放岗位能力 |
| ING-04 | 人工覆盖优先于自动结构化 | 前台看到覆盖结果 |
| ING-05 | 字段覆盖率不足触发卡点 | 进入发布闸门或复核 |

采集测试必须优先覆盖：

- `pipeline.test.js`
- `run-cycle.test.js`
- `position-workbook.test.js`
- `position-overrides.test.js`
- `attachment-only-export.test.js`

## 5. 数据准确性矩阵

数据准确性至少覆盖以下字段和行为：

- 公告时间
- 报名时间
- 附件链接
- 岗位总数
- 专业字段
- 学历字段
- 学位字段
- 去重与更新公告识别

准确性验证口径固定为：

- 抽样比对原公告原文
- 抽样比对岗位表原附件
- 比对结构化字段与原始文本是否一致或可追溯

## 6. SLA 与运营指标

SLA 固定为：

- 新公告正常情况下 1 小时内进前台
- 关键字段错误率目标 1% 以内
- 每来源可见最近成功抓取时间和最近成功发布时间

运营指标最低观察项固定为：

- 当前发布模式
- 当前卡点
- 连续失败次数
- 待复核数量
- 来源是否仍处于仅公告模式

## 7. 回归优先级

回归优先级固定为：

### P0

- 对比分流
- 发布闸门
- 回退

### P1

- 岗位推荐
- 方案保存
- 消息跳转

### P2

- 文案
- 排序
- 展示细节

任何大改如果触碰 P0 范围，必须补或重跑对应回归。

## 8. 验收出口标准

验收通过必须同时满足：

- `Notice`、`PositionBatch`、`PositionNormalized` 术语统一
- `noticeTrust`、`attachment-only`、`待核对` 的语义统一
- 页面名、来源 ID、考试类型、限制值与现有实现一致
- `frontend-contract.md`、`ingest-contract.md`、`test-matrix.md` 三者没有互相矛盾的说明
- `README.md` 和 `mvp-contract.md` 都能作为入口导向分线文档

文档阶段不要求新增自动化测试，但要求：

- 矩阵中的关键场景都能在现有测试目录中找到对应落点
- 后续开发不能绕开本文档自行删减 P0 场景
