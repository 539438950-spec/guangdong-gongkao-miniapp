# 广东公考小程序产品方案

## 1. 产品定位

首版定位为“广东公考信息/选岗工具”，核心不是备考内容，而是：

- 聚合少量高价值官方来源
- 提供稳定、准确、可追溯的数据
- 把岗位检索、岗位对比、订阅追踪做成主能力

不与 `粉笔`、`华图在线`、`高途公考` 正面竞争题库、课程、模考、直播，而是更接近 `公考雷达` 的信息聚合与选岗路径。

差异化重点：

- 更聚焦广东，先把广东来源做深做准
- 把“采集稳定性 + 发布闸门 + 岗位对比”作为产品核心，而不是附加功能

## 2. 竞品对标

### 公考雷达

优势：

- 信息聚合和职位匹配路径成熟
- 提醒和信息追踪能力强
- 用户能快速从公告进入选岗

可借鉴点：

- 从“最新公告”直接进入岗位筛选与对比
- 用订阅、提醒、对比方案降低决策成本
- 用规则解释替代黑盒推荐

### 粉笔 / 华图在线 / 高途公考

优势：

- 题库、课程、模考、训练营完整
- 商业化成熟

不跟随的方向：

- 首版不做题库和课程闭环
- 不做重内容运营
- 不做泛公考培训平台

## 3. MVP 范围

首版只做三条主线：

- 官方公告聚合
- 岗位检索与岗位对比
- 收藏、订阅、消息、浏览记录

明确不做：

- 题库、模考、课程、直播
- 非官方经验社区
- 全国全量覆盖
- 激进的“全网一把抓”

## 4. 首版数据边界

优先来源：

- 广东省人事考试网：`rsks.gd.gov.cn`
- 广东省人社公共服务入口：`ggfw.hrss.gd.gov.cn`
- 国考专题入口：`bm.scs.gov.cn`

上线边界：

- 只有公告和岗位表都能稳定结构化时，才开放岗位对比
- 结构化失败时，允许只展示公告，不开放岗位能力
- 来源连续不稳定时，不带病上线，先进入复核与灰度验证

当前仓库状态：

- `rsks-gd` 已按真实来源接入
- `national-bm` 仍保留为演示源，前台显式标记

## 5. 数据稳定性方案

“稳定且准确”不是单个爬虫脚本，而是一条采集流水线。

当前设计分层：

- `services/ingest`
  - 负责抓取、解析、校验、发布、复核
- `services/api`
  - 负责把同一份快照通过本地 HTTP 或云函数暴露给小程序
- `apps/weapp`
  - 负责公告、岗位、对比、消息、数据状态展示

核心原则：

- 每个来源一个独立 adapter
- 抓取、解析、发布三段解耦
- 原始页面、附件、中间结果全部可追溯
- 校验通过后才进入前台快照
- 新版本异常时继续使用上一稳定版本
- 解析失败进入人工复核队列

前台已体现的稳定性能力：

- 数据源状态页
- 复核中心
- 发布闸门摘要
- 失败类型汇总
- 回退到稳定版本提示
- 来源级 `nextAction`

## 6. 岗位对比设计

岗位对比是首版正式能力，不是附属功能。

规则边界：

- 仅允许同一考试类型内对比
- 单个对比组最多 4 个岗位
- 支持多组方案保存
- 支持从首页、公告、岗位列表、消息、我的等入口进入对比

当前设计重点：

- 对比方案不是一次性临时弹窗，而是可持续维护的“方案”
- 公告、订阅、收藏都能给出“是否适合直接带入对比”的建议
- 同考试类型方案复用优先，满组时进入 review-needed 流程
- 推荐先用规则版，不做 AI 黑盒

规则关注字段：

- 学历
- 学位
- 专业关键词/专业代码
- 基层经历
- 应届限制
- 地区
- 政治面貌
- 其他限制条件

## 7. 当前已实现能力

当前仓库已经落地的主能力包括：

- 首页
  - 最新公告
  - 岗位对比入口
  - 来源状态摘要
  - 发布闸门摘要
  - 复核中心入口
- 公告列表 / 公告详情
  - 公告阶段识别
  - 仅公告与可选岗状态区分
  - 从公告直接进入岗位或对比
- 岗位列表
  - 条件筛选
  - 排序
  - 保存筛选方案
  - 加入对比组
- 岗位对比页
  - 同考试类型对比
  - 规则解释
  - 对比方案复用与替换建议
- 消息中心
  - 订阅命中
  - 进度提醒
  - 快捷进入岗位与对比
- 我的
  - 收藏公告
  - 筛选方案
  - 订阅
  - 对比方案
  - 个人条件
  - 本地/远端 API 切换
- 数据运营页
  - 数据源状态页
  - 复核中心

## 8. 当前最适合继续推进的方向

如果按 MVP 优先级继续做，建议顺序如下：

1. 补更多真实广东来源，但坚持“少来源、高准确”
2. 完善岗位表字段标准化与错误覆盖层
3. 把复核动作真正接到云端管理后台
4. 增加来源灰度开关和发布审计
5. 继续优化岗位对比解释与相似岗位推荐

## 9. 验收标准

产品侧：

- 用户能从公告快速进入岗位筛选和岗位对比
- 没有结构化成功的来源不会误开放选岗入口
- 对比方案可以被保存、复用、删除、重命名

数据侧：

- 新公告正常情况下 1 小时内进入前台
- 来源异常不会拖垮其他来源
- 解析失败会进入复核队列
- 前台可以明确看到最近成功抓取时间、发布时间和发布闸门状态

## 10. 结论

这套产品最合理的方向不是“大而全公考平台”，而是：

- 先做广东
- 先做官方
- 先做稳定
- 先把选岗工具链做透

只要“官方信息聚合 + 稳定结构化 + 岗位对比 + 订阅追踪”这四件事做扎实，就已经有清晰的用户价值和后续扩展空间。
