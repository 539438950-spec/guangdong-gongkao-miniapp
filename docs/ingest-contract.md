# 采集/API 执行基线

本文档服从 [MVP 总契约](./mvp-contract.md)，不定义新的产品模式，只把 `services/ingest/**` 和 `services/api/**` 需要执行的采集、解析、发布、回退和暴露规则写死。  
本文档的直接受众是采集链路、API 服务和数据治理实现者。它不负责页面交互细节，也不负责测试优先级；这两部分分别服从 `frontend-contract.md` 和 `test-matrix.md`。

## 1. 文档目的

采集和 API 实现必须以本文档为唯一执行基线，避免：

- Adapter 之间共用脆弱解析逻辑
- 前台读到未过闸门的候选数据
- 回退规则在不同来源上不一致
- 本地模式和远端模式暴露不同语义的快照

## 2. 来源边界

首版来源只承认以下 3 个：

- `rsks-gd`
- `ggfw-hrss-gd`
- `national-bm`

来源边界固定为：

- `rsks-gd`
  - 模式：`official`
  - 考试类型：`guangdong-provincial`
- `ggfw-hrss-gd`
  - 模式：`official`
  - 考试类型：`guangdong-provincial`
- `national-bm`
  - 模式：`demo`
  - 考试类型：`national`

除非先更新总契约和本文件，不允许新增第 4 个首版来源。

## 3. 采集流水线

采集流水线固定为以下顺序，不允许跳步：

1. 来源调度
2. 原文落盘
3. 附件下载
4. 公告解析
5. 岗位表解析
6. 字段映射
7. 结构化校验
8. 发布
9. 回退 / 复核

各步职责固定为：

- 来源调度
  - 负责轮询周期和来源隔离
- 原文落盘
  - 保存原始 HTML、链接、抓取时间、摘要信息
- 附件下载
  - 负责岗位表及相关附件获取和解压
- 公告解析
  - 输出 `Notice`
- 岗位表解析
  - 输出 `PositionBatch` 候选结果
- 字段映射
  - 输出 `PositionNormalized`
- 结构化校验
  - 决定是否进入发布闸门
- 发布
  - 只导出已通过闸门的快照
- 回退 / 复核
  - 失败时保留旧稳定版，并把异常送入复核中心

## 4. 标准数据模型

必须与共享模型保持一致，统一使用：

- `Notice`
- `PositionBatch`
- `PositionNormalized`

采集层不得发明与这三层并行的新主实体。

固定语义如下：

- `Notice`
  - 官方公告实体或聚合公告实体
- `PositionBatch`
  - 某次岗位表结构化尝试
- `PositionNormalized`
  - 前台消费的标准岗位实体

## 5. Adapter 契约

每个来源必须使用独立 Adapter。

Adapter 最小输入固定为：

- 来源配置
- 抓取时间
- 上一次稳定状态或必要上下文

Adapter 最小输出固定为：

- 原始公告内容或原始列表项
- 原始附件信息
- 可结构化的公告候选
- 可结构化的岗位表候选

Adapter 规则：

- 不允许多个来源共用一套脆弱 DOM 选择器逻辑
- 允许共用通用工具，但不允许把来源特定规则藏进共享工具
- 每个来源的字段缺失、结构变更、附件异常必须能单独告警

## 6. 结构化校验契约

`parseStatus` 允许值和语义固定为：

- `pending`
  - 尚未完成解析
- `parsed`
  - 已完成结构化且通过基本校验
- `failed`
  - 解析失败，不能直接发布岗位能力
- `attachment-only`
  - 抓到了公告和附件，但岗位能力未达发布标准

结构化校验至少覆盖：

- 公告字段完整性
- 岗位行数是否有效
- 表头识别是否成功
- 关键字段覆盖率是否达标
- 关键时间字段是否能抽取
- 是否存在明显重复或异常空列

## 7. 发布闸门契约

发布闸门必须明确判断以下三类结果：

### 7.1 开放岗位能力

同时满足以下条件时才允许：

- 公告抓取成功
- 岗位表下载成功
- 表头识别成功
- 关键字段覆盖率达标
- 批次结构化校验通过

### 7.2 仅公告模式

任一条件成立即可进入：

- 只有公告，没有可用岗位表
- 岗位表解析失败
- 关键字段覆盖率不达标
- 来源被手工覆盖到 `notice-only`

此时必须：

- 允许公告展示
- 禁止岗位对比和推荐
- 前台可见“仅公告模式”或等价说明

### 7.3 进入复核队列

任一条件成立即可进入：

- 新解析版本较稳定版明显退化
- 结构化校验失败且需要人工判断
- 字段映射异常
- 附件下载或解压异常但无法自动修复

## 8. 回退与复核契约

回退规则固定为：

- 新版解析失败时，前台继续使用上一稳定版本
- 单来源失败不影响其他来源
- 回退只影响该来源的当前发布结果，不影响历史记录

复核规则固定为：

- 异常进入复核中心时必须保留候选版本信息
- 复核动作可关闭、重开、批量处理
- 人工覆盖优先于自动结构化结果
- 人工覆盖不能直接篡改原始数据，只能叠加在标准化结果之上

## 9. API 暴露契约

API 层必须明确：

- 小程序只消费“已发布快照”
- 不直接消费候选解析结果
- 复核动作通过 API 回写 ingest 存储
- 本地模式与远端模式读取同一语义快照

当前远端入口固定支持：

- `GET /health`
- `POST /rpc`

本地模式与远端模式的差异只允许存在于“调用方式”，不允许存在于“数据语义”。

## 10. 运维可观测性

每个来源至少要能看到以下字段：

- 最近成功抓取时间
- 最近成功发布时间
- 当前发布模式
- 当前卡点
- 连续失败次数
- 待复核数量

建议同时暴露：

- 当前稳定版本
- 当前候选版本
- 字段覆盖率
- 附件工作表摘要
- 最近一次风险摘要

## 11. 采集验收标准

- 首版来源固定为 `rsks-gd`、`ggfw-hrss-gd`、`national-bm`
- 所有来源都走同一条 9 步流水线
- `parseStatus` 只允许 `pending / parsed / failed / attachment-only`
- 前台只读已发布快照
- 新版失败时前台继续用上一稳定版
- 单来源失败不拖垮其他来源
- `attachment-only` 必须进入仅公告模式
- 每来源都能看到最近成功抓取时间和最近成功发布时间
